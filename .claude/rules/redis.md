你是一个专业的 Go 开发助手，所有 Redis 使用必须严格遵循以下规范。

**注意：本规范适用于项目的 Redis 使用，所有涉及 Redis 操作的场景必须遵守。**

## 总体原则

### 【必须】适用场景
- Redis 仅用于缓存、分布式锁、计数器、热点数据存储等场景

### 【必须】存储限制
- 禁止存储大对象（>1MB），避免阻塞和内存膨胀

### 【必须】过期时间
- 所有 Key 必须设置明确的过期时间（TTL），除非为永久配置项

### 【必须】禁用命令
- 禁止在生产环境使用以下命令：
  - `FLUSHALL`、`FLUSHDB`、`KEYS`、`HGETALL`
  - `EVAL`、`EVALSHA`、`SCRIPT`
  - `SCAN`、`PFMERGE`、`PFCOUNT`

### 【推荐】慎用命令
- 除非必要，否则不要使用以下命令（对大集合运算开销高，可能造成阻塞）：
  - `SUNION` / `SINTER` / `SDIFF`
  - `SUNIONSTORE` / `SINTERSTORE` / `SDIFFSTORE`
  - `ZUNIONSTORE` / `ZINTERSTORE`

## 键命名规范

### 【必须】命名格式

```
{业务模块}:{业务含义}:{唯一标识}
```

### 【必须】命名示例

| 业务场景 | Key 示例 | 说明 |
|----------|----------|------|
| 用户登录 Token | `app:user-token:{user_id}` | 存储用户登录状态 |
| 短信验证码 | `app:user-verify-code:{mobile}` | 手机验证码缓存 |
| 分布式锁 | `billing:lock-order-create:{order_id}` | 创建订单的分布式锁 |

### 【必须】命名规则
- 所有 key 使用**小写字母**，单词间用 `-` 分隔
- key 长度不得超过 **128 个字符**
- 禁止使用中文、空格、特殊字符

## 数据结构使用规范

### 【必须】允许使用的数据结构

| 数据结构 | 使用场景 | 注意事项 |
|----------|----------|----------|
| **String** | 计数、缓存简单对象 | 限制长度在 1MB 以内 |
| **Hash** | 存储对象（如用户信息） | 字段数控制在 1000 以内 |
| **Set** | 标签集合、去重统计 | 限制元素数量在 1000 以内 |

### 【必须】禁止使用的数据结构

| 数据结构 | 说明 |
|----------|------|
| **Sorted Set** | 禁止使用 |
| **List** | 禁止使用 |

## 过期时间规范

### 【必须】TTL 设置标准

| 场景 | TTL 范围 |
|------|----------|
| 普通缓存 | `5min ~ 1h` |
| 临时状态 | `10s ~ 1min` |
| 分布式锁 | `≤ 30s` |

- 所有缓存数据必须设置过期时间，禁止无 TTL 的缓存 key

## 开发注意事项

### 【必须】统一客户端
- 使用统一封装的 Redis 客户端库，禁止直接使用原生 Redis 库

### 【必须】错误处理
- 非关键操作出错，不应影响主路径逻辑
- 代码中应包含熔断和降级机制

### 【必须】功能约束
- 避免使用 Redis 高级功能（如 Lua 脚本、Stream、Pub/Sub 等）
- Redis 锁和定时任务使用统一封装
